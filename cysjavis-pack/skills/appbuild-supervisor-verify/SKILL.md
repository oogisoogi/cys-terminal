# appbuild-supervisor-verify

01~04를 **13항목**으로 기계적으로 검증한다. 근거: GitHub spec-kit `/analyze`(중복·모호·
미명세·정합·커버리지·불일치 6패스)·Kiro EARS·BMad PRD 체크리스트·RTM 추적성·INVEST. 각
발견에 **CRITICAL/HIGH/MED/LOW** 심각도를 단다 — CRITICAL은 빌드를 막는다.

## 13항목 체크리스트

### 일관성 (문서 4종이 서로 맞는가)
1. **고아 기능 없음** — 모든 PRD 기능이 ≥1 화면에 매핑(아니면 `[NON-GOAL]` 명시).
2. **모델 외 데이터 참조 없음** — 화면이 읽고/쓰는 모든 필드·엔티티가 아키텍처 모델에 존재.
3. **부모 없는 작업·미정의 컴포넌트 없음** — 작업마다 대응 요구 존재, 작업이 참조하는 파일·
   컴포넌트가 스펙/플랜에 정의됨.
4. **용어 드리프트 없음** — 같은 개념이 PRD·화면·모델·작업에서 하나의 이름으로(사전 라운드트립).

### 추적성 (요구↔화면↔데이터↔작업 양방향 RTM)
5. **순방향 체인 완결** — 각 요구 → 화면 → 데이터 → 작업 → 수용 테스트까지 끊김 없음.
6. **역방향 고아 0** — 고아 작업/테스트(요구 없음=스코프 크립)·고아 요구(만들었으나 미검증)
   없음. **커버리지% 표**(≥1 작업 가진 요구 비율) 산출.
7. **ID 안정·유일·연속·참조 해소** — 끊긴 FR/S/T ID 없음, 순환·전방 점프 없음.

### 갭 (MVP 단골 누락)
8. **비-해피 상태 명세** — 데이터 화면마다 로딩·빈·에러 상태.
9. **인증/권한 범위 명시** — 누가 보고/하는지, 아니면 `[NON-GOAL for MVP]`로 명시 연기.
10. **입력 검증·엣지 케이스 명세** — 입력별 경계·잘못된 입력·상충 제약(테스트 가능한 EARS 조건).
11. **비기능은 형용사 아닌 수치** — "빠른/안전한"마다 측정 임계, `TODO/???/<placeholder>` 0.

### 품질 (산출물 자체가 좋은가)
12. **작업이 INVEST·수직 슬라이스** — 독립·협상·가치·추정·작게·테스트가능 + 엔드투엔드 데모
    가능(스키마만·UI만인 작업 플래그).
13. **모든 요구가 테스트 가능·수용 기준 보유** — 각 FR에 ≥1 검증 조건, "user-friendly"·
    "적절히" 같은 검증 불가 문구 거부.

## 출력 계약

findings 리포트 `[{item, severity, evidence}]` + 커버리지% 표를 상위 `[[appbuild-supervisor]]`에
반환. CRITICAL/HIGH는 `[[appbuild-supervisor-fix]]`로, 잔여 0이면 `[[appbuild-supervisor-gate]]`로.
