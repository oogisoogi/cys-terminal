# harness-engineering — 하네스 엔지니어링 7기법

> 출처: 외부 수집 지식(YouTube 4JtB_QvKT8w "하네스 엔지니어링" · 빌더조쉬 채널 × 바이브마피아 최수민)
> — **오너 SOT 아님.** 영상에 실제 나온 개념만 자비스 자산으로 매핑했다(환각0: 영상에 없는 기능 날조 금지).

## 핵심 철학

하네스는 **AI 에이전트를 사람 개입 없이 오랫동안 자율주행시키기 위한 장치**다. 에이전트가 그냥
일하면 컨텍스트 크기가 한정돼 많이 못 하는데, 작업을 서브에이전트 단위로 나누고 다른 페르소나로
리뷰시키는 그 세팅이 하네스다.

- **개발 4단계 중 뒤 두 단계에서 강력**: 요구 구체화 → 설계 → **구현 → 검증**. 앞 두 단계는
  사람 의도가 잘 반영돼야 해서 자율주행 의미가 적고 할루시네이션이 쉽다. 하네스는 "이미 잘
  구체화된 것을, 그게 아무리 크든, 틀림없게 구현해 내게 만드는 세팅". **구체화(빡센 기획)는
  사람이 하고, 그 이후 구현·검증은 전부 AI에게 맡긴다.**
- **컨텍스트 = 제한된 리소스**: 쓸모없는 정보는 안 주고 알아야 할 것만 준다. 워커가 의도와
  맞지 않는 작업을 하지 않고 할 일을 클리어하게 끝내게 만드는 것이 목표(산업공학·조직관리적 접근).
- 두 가지 대표 하네스 — ① **구현 하네스**(N5·N6·N7), ② **코드리뷰 하네스**(N1·N4·N3·N7).

## N1 — 세션 포크 → 문서만 회수 (컨텍스트 보존)

요구 구체화·아키텍처 설계로 컨텍스트가 충분히 쌓이면 **세션을 포크**(같은 기억을 가진 동일
세션 복제)한다. 포크된 세션에서는 ① **설계 의도 문서**(의사결정 근거 — 이 작업 한정·아주 자세)와
② **ADR 문서 업데이트**(N2 참조)만 작성하고 포크 세션은 **폐기**한다. 문서(결과)만 남긴다.

- 왜 폐기? 문서화는 새 인사이트 도출이 아니라 기존 인사이트의 문서화일 뿐 → 컨텍스트만 소모.
  포크해서 문서만 쓰고 버리면 **원본(메인) 세션의 컨텍스트가 보존**된다.
- 설계 의도 문서는 에이전트가 쓰면 의도를 못 살릴 때가 있어 **사람이 한 번 검토**한다.
- **자비스 매핑**: master 컨텍스트 60% 도달 전 절약 기법 = 자율주행 축2(SESSION_STATE 최신화→
  CSO가 /clear 집행) 보강. 무거운 문서화(회고·증류·설계의도)는 sub-agent에 위임→문서만 회수.
  회고·증류 인프라는 `javis_reflect.py`, ADR 영속은 `javis_adr.py`(N2).

## N2 — ADR 결정기록 (공용 SOT)

ADR = **Architectural Decision Records**. 설계 결정을 내릴 때마다 문서로 남겨야 나중에
온보딩하는 (사람·에이전트) 개발자가 "이 조직은 이런 기준으로 의사결정한다"를 알 수 있다.

- ADR 문서는 **공용 SOT로 계속 관리**(설계 의도 문서는 작업 한정·일회성과 대비).
- **자비스 매핑**: `javis_adr.py`로 결정을 채번·영속한다.
  ```bash
  python3 "${CYS_PACK_DIR:-$HOME/.cys/pack}/bin/javis_adr.py" add \
      --title "<결정 제목>" --decision "<무엇을 채택했는가>" \
      --context "<배경>" --consequences "<결과·트레이드오프>"
  python3 "${CYS_PACK_DIR:-$HOME/.cys/pack}/bin/javis_adr.py" list          # 전체 결정 한 줄 요약
  python3 "${CYS_PACK_DIR:-$HOME/.cys/pack}/bin/javis_adr.py" relevant --query "<키워드>"  # 이 작업 관련 결정만
  ```
- `relevant`가 N4(압축 서브에이전트)의 **결정론 보조** — 과거 결정 기록 중 이번에 참고할 것만 골라낸다.

## N3 — 평가기준 양방향 (구현 = 리뷰 동일 기준)

평가기준(코드 컨벤션·관련 ADR)을 만들었으면 **구현할 때도 먹이고 리뷰할 때도 똑같이 먹인다.**
코드리뷰가 잘 안 되는 흔한 이유는 ① 컨벤션·ADR을 아예 안 먹이거나 ② 먹여도 방대·장황해
무엇에 포커스할지 흐려지는 것 — 콤팩트한 기준 문서를 만들어 양쪽에 동일 투입하면 퀄리티가 오른다.

- **자비스 매핑**: 위임과 리뷰에 같은 `--success` 기준을 투입한다.
  ```bash
  # 구현 위임 티켓 (성공 기준 = 평가 기준)
  python3 "${CYS_PACK_DIR:-$HOME/.cys/pack}/bin/javis_orchestra.py" task-prompt \
      --task "<T>" --scope "<범위>" --success "<평가 기준>"
  # 리뷰어(agy·codex)에 동일 기준 투입
  python3 "${CYS_PACK_DIR:-$HOME/.cys/pack}/bin/javis_orchestra.py" review-prompt \
      --task "<T>" --scope "<파일>" --success "<평가 기준>"
  ```

## N4 — 압축 서브에이전트 (유사 RAG)

컨벤션 문서·ADR는 계속 업데이트되며 커진다. 그러나 모든 내용이 이번 작업 평가에 참조될 필요는
없다. **이 작업과 관련된 것만 골라내는 서브에이전트** = "압축된 암묵지 — 그것만 보고 해라".

- **복잡한 RAG를 만들 필요 없다.** 서브에이전트만 만들어도 유사 RAG 역할을 꽤 잘 해 준다.
- **자비스 매핑**: `javis_adr.py relevant --query`(N2)가 결정론 보조. soul/CLAUDE/memory/디렉티브가
  비대해졌을 때 "이 작업에 관련된 줄만" 골라 메인 컨텍스트에 주입(능동 모니터링·자율주행 축2 보강).

## N5 — docs-diff (변경된 줄만 참조)

**Task 첫 단계는 무조건 문서 업데이트.** 만들 기능을 말하면 기존 문서가 바뀌어야 하므로,
문서 업데이트를 작업 첫 단계에 박아 두고 전부 갱신한 뒤 **docs-diff 파일**을 만든다. docs-diff =
문서에 변경된 정확한 내용(예: "ADR 문서 248번째 줄에 이런 내용이 추가됐다 / 삭제된 내용은 없다")을
스크립트로 취합한 것. 이후 Phase들이 이 docs-diff를 참조해 **스펙과 어긋난 구현(spec-drift)을 차단**한다.

- 스펙 기반 작업 계획을 에이전트가 제멋대로 해석해 스펙과 어긋나게 구현할 우려를 막는다.
- 서비스가 커질수록 스펙 문서도 많고 길어진다 → "문서 바뀌었어" 언급으론 불충분. **추가된 줄만
  들어가니 강조가 확실**해진다. 첫 단계에 넣는 것이 정확도 컨트롤에 매우 중요.
- **자비스 매핑**: `javis_docsdiff.py`로 변경 줄만 추출한다.
  ```bash
  python3 "${CYS_PACK_DIR:-$HOME/.cys/pack}/bin/javis_docsdiff.py" \
      --paths <문서들> [--base <git-ref>] [--out docs-diff.md]
  ```

## N6 — Task / Phase 순차 위임 (메인 컨텍스트 보존)

- **Task** = 작업 하나 통째. **Phase** = 그 작업을 마치기 위해 나눈 단계들(보통 10개, 20개를
  넘기도). 각 Phase는 **독립 세션이 "이것만 보고도" 의도 유실 없이 완수**하도록 **자기완결**로
  작성한다 — 스펙 경로 문서·docs-diff(N5)·직전 Phase 산출물을 잘 참조시키고, 구체적 작업 지침과
  검증 방법까지 포함한다.
- **순차(직렬) 실행**: 구현은 단순 스크립트 호출만으로도 잘 된다. 각 Phase의 구체적 구현
  컨텍스트는 하위 일꾼 세션에 넘기고, 메인 세션은 사용자 의도·"이 작업이 어떻게 되어야 하는가"를
  기억하는 데 집중 → **구현을 끝낸 뒤에도 메인 컨텍스트가 20%도 소모 안 됨**(버그 논의 시 메인이
  클리어해서 답변 잘 줌). 중간에 깨지면 잘된 지점까지 기억해 두고 복구·재개.
- ★**자비스 매핑 — claude -p raw subprocess는 띄우지 않는다**: 영상은 파이썬 스크립트가 헤드리스
  `claude -p`를 서브프로세스로 순차 호출하지만, 자비스는 `harness-creator`의 PROMPT_RUNNER_ABSENT
  철학·자원 거버넌스(로컬 서버·load 마비 방지)와 충돌하므로 **회피**한다. 대신 `phase-plan`으로
  자기완결 티켓들을 만들고, **Workflow pipeline 또는 cys 워커 순차 위임**으로 실행한다.
  ```bash
  python3 "${CYS_PACK_DIR:-$HOME/.cys/pack}/bin/javis_orchestra.py" phase-plan \
      --task "<T>" --phases "설계;구현;검증" --scope "<범위>" [--success "<기준>"] [--to worker]
  ```
  각 Phase 티켓은 `build_task_ticket`을 재사용해 **절대 강조 4규칙**(품질 절대우선·할루시네이션
  방지·의도 합의(grill-me)·요약 압축 금지)을 자동 포함하고, "이 Phase만 독립 실행" 자기완결 지시가 붙는다.

## N7 — 코드리뷰 하네스 + 에이전트 깃헙 계정

리뷰는 웬만한 상황에서 가장 큰 병목(구현은 됐는데 리뷰를 못 받아 머지 블로킹) → 최적화 효용이 크다.

- 요구 구체화·아키텍처 설계는 여전히 사람이 → 컨텍스트가 충분히 쌓이면 N1 포크 → 설계의도·ADR
  문서 작성 후 포크 폐기 → 원본 세션이 N4 압축 서브에이전트로 이번 작업 평가기준만 추려 → N3로
  구현·리뷰에 동일 투입.
- **PR = 인간(최종 검수자)이 보기 편한 로그·보고서.** 머지 자체가 목적이 아니라, 커밋·대화
  내역을 나중에 참조하기 좋은 형태로 쌓아 두는 문서. 자동 리뷰(CodeRabbit류)가 PR에 리뷰를
  남기면 **에이전트가 반영할 만한 리뷰를 자동으로 읽어 자동 반영**("반영했습니다" 댓글 →
  리뷰봇 "확인했습니다" → 마무리). 사소한 구현 개선점은 사람과 논의하지 않고 에이전트끼리 처리.
- ★**denylist — 오너 승인 후 활성화**: 영상은 **에이전트 전용 깃헙 계정**(본인 계정 아님)을
  만들어 실제 `git push`·PR 생성·**머지 자동화**까지 돌린다. 자비스에서 **외부 깃헙 계정 생성·
  git push·gh release·머지(외부발행=비가역)는 denylist** — 이 스킬은 **기법·운영 가이드만**
  기술하고, **실제 활성화는 오너 명시 승인 후에만** 한다(로컬 커밋은 가역=허용, 외부발행은 정지).

## 자비스 기존 자산 연결 (중복 아님 · 보완)

- **harness-creator** — 도메인 한 문장 → 풀스택 Claude 프리미티브 하네스를 **코드 산출물로
  생성·검증·진화**하는 메타스킬. 본 스킬은 그 하네스를 **운영하는 7기법 원리**를 제공.
- **self-correction-loops** — 객관 eval을 LOCK하고 producer≠evaluator로 채점하는 **적대적
