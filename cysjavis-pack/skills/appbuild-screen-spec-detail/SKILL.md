# appbuild-screen-spec-detail

각 화면을 구현 가능한 수준으로 상세화한다. **빠지기 쉬운 상태(빈·에러·로딩)를 빠뜨리지 않는
것**이 핵심 — 갭 검증의 단골 결함이다.

## 화면별 명세 항목

- **레이아웃**: 주요 영역·컴포넌트 배치.
- **상태 전부**: 정상 · **빈 상태** · **에러 상태** · **로딩** (+ 권한/미인증이 있으면 그 상태).
- **컴포넌트**: 재사용 컴포넌트 목록·동작.
- **데이터**: 이 화면이 읽고/쓰는 데이터 — `03-architecture.md` 모델의 엔티티·필드와 일치.
- **액션**: 사용자 동작 → 결과(데이터 변경·이동).

## 절차

1. **화면별 상세** → 검증: 흐름 단계의 각 화면에 위 항목을 채운다. **상태 4종을 의식적으로
   확인**(빈/에러/로딩 누락 0).
2. **데이터 정합** → 검증: 화면이 참조하는 데이터가 전부 아키텍처 모델에 존재하는지(모델에
   없는 데이터 참조 0).

## 출력 계약

`.appbuild/02-screens.md`의 화면별 상세 섹션. 상위 `[[appbuild-screen-spec]]`로 반환. 상태
누락·모델 외 데이터 참조는 보고·보강(감독관 갭·추적성 검증의 대상).
